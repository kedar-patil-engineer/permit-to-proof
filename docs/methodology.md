# Evaluation methodology

This document defines how the artifact's evaluation harness (`app/eval`,
`evaluate.py`) produces the numbers the paper reports (master document Part A5),
and the assumptions and limits a reviewer should know. It is the methodological
companion to the code; the code is the source of truth.

## The four metrics

Run the pipeline twice over a permit — verification **ON** and **OFF** — on the
same extraction, then grade against a gold answer key.

### 1. Extraction quality (precision, recall, F1)

Each extracted obligation is greedily aligned one-to-one to at most one gold
obligation (`app/eval/metrics.py: match_extractions`). Two obligations match
when, after the verifier's own normalization:

* parameters are the same canonical pollutant (or, for narrative obligations
  with no parameter, their descriptions have difflib similarity ≥ 0.60);
* numeric values agree within 1% relative tolerance (so 30 vs 30.0 matches but
  250 vs 30 does not);
* units are equivalent when a numeric value is present;
* operator families agree (`<=`/`<` vs `>=`/`>` vs `=` vs range), when the gold
  specifies an operator.

A claimed gold is a true positive (TP); an extraction that claims none is a
false positive (FP); an unclaimed gold is a false negative (FN). A duplicate
extraction cannot re-claim an already-claimed gold, so it is correctly an FP.
Precision = TP/(TP+FP), Recall = TP/(TP+FN), F1 = 2PR/(P+R).

### 2. Verification lift (the headline result)

An **error** is any extraction that matches no gold obligation (hallucinated,
mislabeled, mis-valued, wrong-unit, flipped-operator, or duplicate). The layer
**catches** an error when it routes it to a human (status FLAGGED or
NEEDS_REVIEW). For ON and OFF separately we form the error-detection confusion
matrix (caught/silent errors, false alarms) and compute error-detection recall
and precision. Because the OFF baseline trusts everything, its recall is 0 by
construction, so **lift = error-detection recall (ON)**. We report the absolute
counts and the number of true errors alongside the rate, because on a small or
synthetic set the denominator is small and must be disclosed.

### 3. Confidence calibration (ECE, reliability diagram)

Confidence is binned into M equal-width bins over [0, 1]. For each bin we
compute empirical accuracy (fraction that matched gold) and mean predicted
confidence. ECE = sum over bins of (bin_size / n) * |accuracy - confidence|;
MCE is the worst bin gap. We compute calibration for both the final calibrated
confidence and the backend's raw self-reported confidence so the paper can
contrast them. The reliability diagram plots accuracy against mean confidence
with the perfect-calibration diagonal.

### 4. Selective-prediction trade-off

Sweeping the routing threshold, for each threshold we measure the automation
rate (fraction auto-accepted to VERIFIED) and the auto-accept accuracy (how many
auto-accepted obligations actually match gold). This is the
automation-versus-human-effort curve. We report the operating point: the most
automation achievable while keeping auto-accept accuracy at or above a target
(default 0.95). Obligations that fail an error-level check never auto-accept at
any threshold, which is the intended behavior.

## The gold answer key

The metrics are only as trustworthy as the gold set.

* The **real** answer key (master document A5.4) must be built by a domain
  expert on **real** permits, not generated by a model — otherwise the AI is
  graded against another AI. Target a few hundred labeled obligations across
  20–40 permits; report inter-annotator agreement (for example Cohen's kappa) if
  a second annotator is available. Record provenance as `EXPERT_SINGLE` or
  `EXPERT_ADJUDICATED`.
* The **bundled** gold set (`sample_data/gold/sample_permit.json`) is
  `ILLUSTRATIVE_AUTHOR_KNOWN`: the true obligations of the synthetic permit,
  known because the project authored the PDF. It exists only to exercise the
  harness end to end and to let anyone reproduce the pipeline self-test. It is
  not a substitute for the expert key, and every figure and table labels it as
  illustrative.

## Building the gold key: tooling

The repository ships the toolkit that turns hand labeling into a clean,
reproducible key, so the only irreducible step is the expert reading the permits.

* `annotation/PROTOCOL.md` is the labeling protocol: what counts as an
  obligation, field-by-field instructions, the matching rules fixed in advance,
  and the two-annotator process.
* `annotation/gold_template.csv` is the fillable template (one row per
  obligation), openable in Excel or Google Sheets.
* `python -m app.eval.annotate` converts a filled template into the validated
  gold JSON the harness reads (and can export an existing key back to the
  template for review). It fails loud on a malformed row rather than guessing.
* `python -m app.eval.agreement` computes inter-annotator agreement between two
  independently labeled keys: span-level agreement (a pairwise F1), per-field
  exact agreement on matched obligations, and Cohen's kappa on the operator
  family and the parameter.

The labeling itself is done by a domain expert and must not be generated by a
model. Once the key exists, `python evaluate.py` produces every table and figure.

## Threats to validity

* **Answer-key scope.** Accuracy numbers are bounded by the size and coverage of
  the gold set; the bundled illustrative set is tiny (15 obligations on one
  synthetic permit) and is for self-test only.
* **Synthetic data.** The bundled permit is fictional. Real permits vary in
  layout, OCR quality, and phrasing; extraction and grounding behavior on real
  PDFs must be measured separately.
* **Model-version drift.** Hosted-model results depend on the exact model
  version; the artifact records the backend and model so runs are attributable.
  The Mock backend is deterministic and version-free.
* **Plausibility envelopes.** The domain ranges (see `domain_knowledge.md`) are
  engineering judgment, not regulatory limits; they bound plausibility, not
  compliance.

## Data and Code Availability

The code is this repository (MIT licensed). The bundled sample permit is
synthetic and fully included; it contains no real or copyrighted data. The
illustrative gold set is included. The expert answer key and any real permits
used for the paper's measured evaluation are described here as a protocol and
are released separately as a research asset (master document A3, A5.4). All
reported metrics are reproducible with `python evaluate.py` plus the relevant
gold set.
