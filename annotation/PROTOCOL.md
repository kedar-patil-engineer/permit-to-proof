# Gold answer key annotation protocol

This protocol turns a stack of real permits into the expert gold answer key the
paper requires (master document A5.4). It exists so that two people labeling
independently produce comparable keys, so inter-annotator agreement can be
computed, and so the matching rules are fixed before anyone looks at a result.

The labeling itself must be done by a person who understands environmental
compliance. It must NOT be generated by a language model: grading the system
against model-written labels measures whether two models agree, not whether
either is correct, and that is a reject at any serious venue.

## The workflow at a glance

1. Collect real permits. Title V air permits and NPDES water permits are public.
   Sources: EPA ECHO (echo.epa.gov), EPA NPDES databases, and state portals (for
   example the California Air Resources Board and the State Water Resources
   Control Board). Save each as a PDF under `sample_data/permits/`.
2. For each permit, copy `annotation/gold_template.csv` to
   `annotation/<permit_stem>.csv` and fill one row per true obligation while
   reading the permit. Delete the two EXAMPLE rows.
3. Convert the filled template to the gold JSON the harness reads:
   `python -m app.eval.annotate --csv annotation/<stem>.csv --permit-id <ID>
   --source-pdf sample_data/permits/<stem>.pdf --provenance EXPERT_SINGLE
   --labeler "Your Name" --out sample_data/gold/<stem>.json`
4. (Recommended) A second annotator labels a subset independently. Compute
   agreement: `python -m app.eval.agreement --a A.json --b B.json --out iaa.json`.
   Resolve disagreements by discussion and mark the agreed key
   `EXPERT_ADJUDICATED`.
5. Run the evaluation; the tables and figures fill themselves:
   `python evaluate.py --pdf sample_data/permits/<stem>.pdf
   --gold sample_data/gold/<stem>.json --backend OpenAI --out eval_out/<stem>`

## What counts as an obligation

An obligation is a single, distinct requirement the permit places on the
permittee. Label it when the permit states a thing the facility must do or must
not do. Typical kinds:

* A numeric limit: a parameter must stay at or below (or above, or within) a
  value, in a unit, for example "NOx shall not exceed 30 ppm".
* A monitoring requirement: a parameter must be measured at some frequency.
* A reporting or recordkeeping requirement: a report due by a deadline, records
  kept for a period.

Do not label: definitions, boilerplate, the facility address, statements of
authority, or pure narrative that imposes no duty. When a single sentence
imposes two duties (a limit AND a monitoring frequency), you may record them as
one obligation if they are inseparable, or as two if each stands alone; be
consistent and note your choice.

## Field by field

One row per obligation. Leave a cell blank when the field does not apply.

| Column | Required | What to write |
|---|---|---|
| `gold_id` | yes | A stable id you assign, for example G001, G002 (unique within the permit) |
| `parameter` | for limits | The pollutant or parameter as written (NOx, pH, BOD5, Flow). Blank for a pure reporting obligation |
| `limit_value` | for numeric limits | The numeric value only (30, 0.30, 6.0). Blank if there is no number |
| `limit_unit` | for numeric limits | The unit as written (ppm, mg/L, lb/MMBtu, s.u., deg C, MGD, cfu/100 mL) |
| `operator` | for numeric limits | One of `<=` `<` `>=` `>` `=` `range`. Use `<=` for "shall not exceed", `>=` for "shall not fall below", `range` for "between X and Y" |
| `frequency` | if stated | continuous, daily, weekly, monthly, quarterly, annual, and so on |
| `deadline` | if stated | The reporting deadline or schedule, for example "within 28 days" |
| `citation` | recommended | The permit section or regulatory reference (Condition 3.1, 40 CFR 60.44c) |
| `source_segment_id` | optional | Leave blank; informational only |
| `is_obligation` | optional | TRUE by default. Use FALSE only to record a span you deliberately judged NOT an obligation |
| `description` | yes | The requirement in plain words, close to the permit's own sentence. Used to match narrative obligations that have no parameter |

Record values exactly as the permit states them. Do not convert units, do not
round, do not infer a limit the permit does not state.

## The matching rules (fixed before you see any result)

These are the rules the grader uses to decide when a system-extracted
obligation matches a gold obligation. They are fixed here, in advance, so the
evaluation cannot be tuned to flatter the result. They mirror
`app/eval/metrics.py`.

* Parameter: same canonical parameter (PM10 and PM are the same family; NOx and
  nitrogen oxides are the same). For obligations with no parameter (narrative
  reporting), match on description similarity at or above 0.60.
* Numeric value: agree within 1 percent relative tolerance (so 30 and 30.0
  match; 250 and 30 do not).
* Unit: equivalent after normalization, when a numeric value is present.
* Operator: same family (`<=`/`<`, `>=`/`>`, `=`, `range`), when the gold
  specifies an operator.

A matched gold is a true positive; an unmatched system output is a false
positive; an unclaimed gold is a false negative.

## Provenance labels

Set the gold set's `label_provenance` honestly:

* `EXPERT_SINGLE` — labeled by one domain expert.
* `EXPERT_ADJUDICATED` — labeled by two or more, disagreements resolved.
* `ILLUSTRATIVE_AUTHOR_KNOWN` — reserved for the synthetic self-test only; never
  use this for a real permit key.

The provenance is printed on every figure and table the harness produces, so the
paper's measured numbers are never confused with the synthetic self-test.

## Inter-annotator agreement

When two people label the same permit independently, run `app.eval.agreement`.
It aligns the two keys with the same matching rule above and reports:

* span-level agreement (how many obligations both found, and each found alone),
  summarized as a pairwise F1;
* per-field exact agreement on the matched obligations;
* Cohen's kappa on the operator family and on the parameter, correcting for
  chance agreement.

Report these in the paper's evaluation section. Aim to label enough obligations
that agreement and, especially, the calibration figure are meaningful: a few
hundred obligations across 20 to 40 permits is the target; calibration on a few
dozen obligations is noise.
